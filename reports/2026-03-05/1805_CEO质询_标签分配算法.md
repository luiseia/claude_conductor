## 二、标签分配算法分析

### 当前分配逻辑 (已读源码)

**数据流**: `generate_occ_flow_labels.py:245-294` -> `git_occ_head.py:361-522`

1. **Grid 分配**: 将 3D 框投影到 BEV，计算 2D 包围盒覆盖的 grid cells。判定标准是"grid cell 中心点是否落在 2D 包围盒内"。
2. **深度排序**: 每个 cell 内按深度排序，取最近 3 个物体填入 3 个 slot。
3. **权重**: IBW (Inverse Breadth Weighting) + pos_multiplier=20x 拉升前景权重。

### 分析 1: Grid 分配逻辑审计

**现状问题**:
- 当前使用 2D axis-aligned bounding box (AABB) 判定覆盖。对于旋转的 3D 框，AABB 会显著过估覆盖面积（典型的 45° 旋转车辆，AABB 面积约为真实投影的 1.4-2x）。
- 过估覆盖 -> 更多 cell 被分配为正样本 -> 边缘 cell 的标签质量差（GT 回归目标不准确）-> 训练噪声增加

**可行改进**:
- **方案 A (简单)**: 使用旋转 BBox 的实际多边形覆盖判定 (`shapely.Polygon` 或自写光栅化)。实现成本低（~50 行），但需要在 pipeline 中增加角度信息传递。
- **方案 B (激进)**: 只分配"3D 框中心所在 cell + 直接相邻 cells"，不做全覆盖。这大幅减少正样本数量但提高标签精度。

**推荐**: 方案 A。当前精度问题的部分原因可能来自边缘 cell 的低质量标签。
**优先级**: 中。应在 Plan D 稳定后尝试，不适合在 loss 调参阶段同时修改。
**预计工作量**: 修改 `generate_occ_flow_labels.py` 中的 `_compute_fine_cell_ids` 函数，约 50-80 行代码。

### 分析 2: 2D-3D 视觉对齐

**CEO 思路**: 利用 3D 框对应的 2D BBox，借助 2D 检测的成熟特征来辅助 grid 真值分配。

**可行性评估**:
- nuScenes 数据集有 3D 标注但**不直接提供 2D BBox**。需要将 3D 框投影到各相机视图得到 2D 框。nuScenes-devkit 支持此操作。
- GiT 架构的 encoder 已经提取了多尺度特征。可以在 encoder 输出上做 2D RoI 特征提取，然后注入 BEV decoder。
- 但这改变了模型架构（需要 RoI pooling 或 cross-attention），不是简单的标签层面改动。

**两层实现思路**:
1. **标签层面** (简单): 用 2D 投影框面积作为 cell 权重调制。中心 cell 权重高，边缘 cell 权重低。不改模型架构。
2. **特征层面** (复杂): 用 2D RoI 特征做 cross-attention 引导 BEV prediction。需要改模型。

**推荐**: 先做标签层面（权重调制），与"分析 3: Center/Around"策略协同。
**优先级**: 中低。当前瓶颈是 loss 平衡，不是标签质量。
**预计工作量**: 标签层面 ~30 行；特征层面 ~300+ 行 + 架构调试。

### 分析 3: Center/Around 采样策略

**CEO 思路**: 将 GT 中心所在 cell 定义为 center，其余覆盖 cells 为 around。未来可用 3D position ray 做 prompt。

**可行性评估**:
- 当前代码已经计算了 `cx_img, cy_img`（几何中心坐标），用于兜底策略 (line 289-292)。
- 扩展为 center/around 只需在 `cell_to_gts` 中增加一个 `is_center` 标记。

**实现方案**:
```python
# 在 generate_occ_flow_labels.py 中
center_cell = center_r * GRID_FINE_W + center_c
for cell_id in cell_ids:
    info['cell_type'][cell_id] = 'center' if cell_id == center_cell else 'around'
```

**收益分析**:
1. **权重分化**: center cell 给更高权重（这是物体"最可靠"的位置），around cell 给较低权重。这自然解决了大物体覆盖过多 cell 导致的噪声问题。
2. **3D Ray Prompt**: center cell 可以编码物体的 3D 中心位置 ray (从相机到 BEV 中心的方向向量)，作为 decoder 的位置提示。这为未来的 query-based detection 架构铺路。
3. **与 IBW 互补**: 当前 IBW 让大物体每个 cell 权重降低 (1/N)。center/around 进一步分化：center cell 保持高权重，around cells 大幅降低。

**推荐**: 高优先级。实现简单，与当前架构兼容，且为未来扩展打下基础。
**优先级**: 高。可以在 Plan D 阶段引入。
**预计工作量**: pipeline 修改 ~30 行，head 修改 ~20 行。

### 分析 4: Token 与类别优化

#### 4a: 合并 Marker 和 Class

**当前**: 每个 slot 10 tokens: [marker, class, gx, gy, dx, dy, w, h, th_g, th_f]
- marker: NEAR/MID/FAR/END (4 个 ID)
- class: car/truck/bus/trailer (4 个 ID)

**合并方案**: 将 marker+class 合为一个 token: NEAR_car, NEAR_truck, ..., FAR_trailer, END (4*4+1=17 个 ID)
- 每个 slot 减少到 9 tokens
- 3 slots = 27 tokens (从 30 减少到 27)

**收益**:
- 减少序列长度 10%，加速训练和推理
- 减少自回归解码步数 -> 减少错误累积
- marker 和 class 的语义本质上是一体的（"这里有一辆近处的车"）

**风险**:
- 词表从 4+4=8 个分类 token 扩展到 17 个。如果某些组合（如 FAR_trailer）极稀少，可能学不好。
- 需要同时修改 encoder 词表、decoder 推理、评估代码。

**推荐**: 值得做但需要仔细实现。
**优先级**: 中。在 loss 平衡问题解决后再尝试。
**预计工作量**: 词表重定义 + head/decoder 修改，~200 行。

#### 4b: Construction Vehicle 独立为第五类

**当前**: truck 类包含了 `truck` + `construction_vehicle`。
- nuScenes-mini 的 GT 中两者占比如何需要统计
- construction_vehicle 体型差异大（吊车 vs 推土机 vs 卡车），类内方差远高于 truck

**收益**:
- 降低 truck 类内方差 -> 回归目标更一致 -> truck 学习更稳定
- 解释了为什么 truck 一直是最不稳定的类别：类内混杂了不同形态的车辆

**风险**:
- 增加一个类别 -> 词表增加 1 个 token -> 数据更分散
- 如果 construction_vehicle 在 mini 集中样本极少，可能无法学习
- 需要修改 pipeline 的标签映射

**推荐**: 先统计 nuScenes-mini 中 construction_vehicle 的数量。如果 > 50 个实例，值得分离。
**优先级**: 中。
**预计工作量**: config 修改 + pipeline 标签映射，~30 行。
